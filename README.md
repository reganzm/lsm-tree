### 最简单的数据库

- 提供数据写入set
- 提供数据查询get

优点：数据追加拥有极致的性能，实现简单
缺点：数据读取时间随着数据量线性增长：o(n)；不支持范围查询，需要按key遍历；占用大量磁盘空间，容易磁盘资源耗尽；

### 优化查询性能1:内存中的hashmap索引

- 在不影响原始数据的情况下，提供额外的数据结构hashmap索引，key为db中的key，value为key在db中的offset
- 查询key时，先在hashmap中找到key的offset，然后o(1)复杂度定位到value的偏移量，取出数据

优点：实现简单，对于单条key，可以直接定位到数据在db文件中的offset，直接取出数据，而不是按o(n)遍历
缺点：不支持范围查询；当key非常多时，存在内存中的hashmap索引可能造成oom；hashmap也存在reset capacity的可能，潜在的性能开销；当系统崩溃时，hashmap索引容易丢失，系统恢复需要从原始数据重建索引，当数据量特别大时，系统恢复将非常慢；

这种数据库，显然是不能提供商用的，只能是实验室里面的试验品。

### LSM-Tree 日志合并

- 要杜绝持续增长的磁盘，需要定时合并文件中相同的key；要支持范围查询，文件中的数据最好按key有序存储；避免系统崩溃后重建索引，需要对索引进行持久化处理。
- 要保证写入的数据有序，需要在内存中维护一个有序的数据结构，如红黑树，trie等；数据可以以极好的性能按顺序写入到内存中该数据结构中。
- 为了避免内存中的这个有序的数据结构因系统奔溃而丢失，需要实时的将该结构，以追加的形式立即写入的Log日志文件中；不一定需要保证Log中数据有序，Log中的日志主要用于系统崩溃后，重建内存中的有序的数据结构。
- 按时或按量将内存中的有序数据结构写入到db的一个segments中；因为segments是有序的，所以只需要将segments的稀疏的记录作为索引即可，这形成了一种稀疏的索引结构：sparse index。
- 数据读取，首先从文件中有序的数据结构中查询数据，如果查询到直接返回；如果没有查询到，则查询最新的一个segments的索引，通过索引快速定位到segments位置，遍历segments中的数据，进行查找；如果没有找到，则遍历次新的segments，以此类推，直到找出该条数据为止，潜在的问题是，可能会遍历完所有的segments才找到或则找不到数据，带来极差的性能。
- 为了解决遍历整个db查找数据带来的极差的性能体验，引入基于bitmap布隆的过滤器（n*filter），快速判断数据是否存在于segments中，当新增或删除数据，需要修改布隆过滤器。
- 稀疏索引和filter都维护在segments中，且segments中key唯一。为了避免系统崩溃后恢复重建布隆过滤器和索引，需要将他们写到Log文件中，以便进行崩溃恢复。
- 后台线程，在后端合并不同的segments为新的segments；合并过程中，老的segments对外提供读取服务；合并完成后，新产生的segments对外服务，同时删除老的segments，合并可以基于归并排序算法，合并后的数据仍然是有序的。产生新的segments后，生成新的索引，并对外提供服务

优点：避免了数据持续增长而占满磁盘；数据是排序的，支持范围查询；提供了redo日志，用于崩溃恢复；实现了稀疏索引，便于快速定位数据；具有良好的写入吞吐； RocksDB , Cassandra, Scylla, and HBase

### segments合并策略

- 基于大小的合并策略：会产生大文件，优先合并最新写入的小segments，适合多写少读的场景。
- 基于层级的合并策略：每个文件大小固定，最新的数据写入到L0,按大小分裂到L1，L2等，适合多读少写的场景。
